#### 成员 D（后端+测试）—— 安全 + 性能 PPT + Q&A 初稿

**目标**：写出"安全性设计"和"性能设计"两章 PPT，整理 Q1-Q10 标准回答。

##### Part 1：安全性具体设计（对应 Q8）

**主要威胁与应对（用 STRIDE 模型组织）：**

| 威胁类别 | 具体风险 | 应对措施 |
|---------|---------|---------|
| **认证与授权** | 未授权用户访问别人文档 | JWT Token 鉴权 + 文档与用户 ID 绑定，查询时强制按 user_id 过滤 |
| **数据隔离** | 不同用户的向量数据混在一起 | ChromaDB 按 user_id 分 collection，查询时只查自己的 collection |
| **文件上传攻击** | 恶意文件、超大文件、可执行文件伪装 | 文件类型白名单（仅 .pdf/.docx/.md）+ MIME 验证 + 文件大小限制（≤20MB）+ 病毒扫描（可选）|
| **Prompt 注入** | 用户在文档或问题里植入"忽略上面所有指令"等攻击 | 系统 Prompt 与用户输入严格分离 + 输出过滤敏感指令模式 |
| **API 滥用** | 用户疯狂调用 LLM API 烧钱 | 限流（每用户每分钟 10 次问答）+ 单文档上传限额 |
| **数据泄露** | LLM 提供商记录用户文档内容 | 提示用户敏感数据请使用本地部署 / 选择 ZDR（零数据保留）API |
| **传输安全** | 中间人窃听 | 全站 HTTPS（TLS 1.2+）|
| **存储安全** | 服务器被入侵导致数据泄露 | 静态加密（数据库加密、文件存储加密）+ 定期备份 |
| **依赖安全** | 第三方库有漏洞 | 用 `pip-audit` / `npm audit` 定期扫描，CI 中集成 |
| **日志与审计** | 无法追踪可疑行为 | 关键操作日志（上传、删除、登录）+ 异常告警 |

##### Part 2：性能具体设计（对应 Q9、Q10）

**Q9 — 峰值需求/容量预估：**

| 指标 | 当前（开发） | 演示期 | 6 个月后（假设上线推广） |
|------|-------------|--------|------------------------|
| 并发用户 | 1-5 | 10 | 100-500 |
| 每日提问量 | 50 | 500 | 50,000 |
| 文档总数 | 20 | 200 | 100,000 |
| 单文档大小 | ≤ 20MB | ≤ 20MB | ≤ 50MB |
| 向量库容量 | < 10 万 chunk | < 100 万 | > 5000 万 |

**Q10 — 性能瓶颈与改进：**

| 瓶颈 | 当前问题 | 改进方案 |
|------|---------|---------|
| LLM API 延迟 | 单次调用 3-8 秒，是最大瓶颈 | 流式输出（边生成边显示，感知延迟 ↓ 50%）+ 简单问题用更快的小模型 + 同问题缓存 |
| 文档解析慢 | 大 PDF 解析耗时 30 秒+ | 异步任务队列（Celery / FastAPI BackgroundTasks）+ 进度反馈 + 解析结果缓存 |
| 向量检索慢 | 数据量上去后线性扫描慢 | ChromaDB 默认 HNSW 索引已较优；进一步可换 Milvus / Qdrant 支持分布式 |
| 重复 Embedding | 同一文档反复 Embed 浪费 token | 文档内容 hash → 已 embed 则跳过 |
| 数据库连接 | SQLite 高并发写入会锁 | 高并发场景迁移到 PostgreSQL + 读写分离 |
| 前端首屏慢 | React 包体积大 | Code splitting + CDN + Service Worker 缓存 |

**性能指标承诺：**
- 问答首字响应 ≤ 2 秒，完整回答 ≤ 10 秒
- 文档上传 + 索引：10 页 PDF ≤ 30 秒
- 同时支持 100 并发用户无降级

## 三、Q1-Q10 答辩话术

### Q1：你们是否为项目前端设计了纸质原型/线框图/高保真模型/可操作原型？

**话术**：
是的，我们做了完整的四阶段原型。
- **纸质原型**：项目启动时，我们用便利贴在白板上画了核心交互流程
- **线框图**：用 Excalidraw 画了 3 张低保真线框图，包括登录页、主聊天页、文档管理页
- **高保真模型**：基于线框图细化了配色、字体、组件细节
- **可操作原型**：我们用 React + TailwindCSS 实现了一个可交互的前端原型，可以现场演示——这也是您今天看到的这个界面

**演示时机**：现场切到 A 的前端项目，点几下展示。

### Q2：你们前端的 MVC 组件是什么？

**话术**：
前端框架使用 React，严格说 React 是组件化架构而非传统 MVC，但我们按 MVC 思路做了关注点分离：
- **Model（数据层）**：包括两部分——`services/api.ts` 封装与后端的所有 HTTP 通信；用 Zustand 管理全局状态（当前会话、文档列表、用户信息）
- **View（视图层）**：纯展示组件，比如 `<ChatBox>`、`<DocumentList>`、`<CitationPanel>`、`<MessageBubble>`，只负责渲染
- **Controller（控制层）**：用自定义 Hooks 实现，比如 `useChat()` 处理消息发送、流式接收、错误处理；`useDocuments()` 处理文档上传、删除、列表刷新

这样视图组件保持纯净，业务逻辑集中在 Hooks，数据请求统一在 service 层，便于测试和维护。

### Q3：你们评估/测试项目用户体验的计划是什么？

**话术**：
我们计划在 Sprint 3 末进行第一轮 UX 测试，方法包括：

1. **可用性测试**：邀请 5-8 名目标用户（学生 + 研究生），让他们完成 3 个核心任务：
   - 任务 A：上传一个 PDF 并提一个问题
   - 任务 B：基于第一个回答进行追问
   - 任务 C：找到并查看回答的引用来源

2. **量化指标**：
   - 任务完成率 ≥ 90%
   - SUS 量表得分 ≥ 70 分（业界"良好"线）
   - 提问到理解回答的平均时间 ≤ 30 秒

3. **质性反馈**：测试后做半结构化访谈，记录卡点、困惑、建议

4. **A/B 测试（可选）**：对引用展示样式做小规模 A/B 对照

测试结果会驱动下一轮设计迭代。

### Q4：你们项目的组件图是怎样的？

**话术**：
我们的系统分为五层：

1. **前端层（React）**：文档上传、聊天界面、引用展示三个核心组件
2. **后端层（FastAPI）**：暴露三组 API：`/upload`、`/chat`、`/documents`
3. **AI 逻辑层（LangChain）**：包括文档解析器、RAG Chain（检索+生成）、对话记忆管理三个核心组件
4. **数据层**：ChromaDB 存向量索引，SQLite 存对话历史，本地文件系统存原始文档
5. **外部服务**：调用 OpenAI 或 Claude API 完成 Embedding 生成和最终回答生成

组件之间通过明确的接口通信：前端通过 REST API 调后端，后端通过函数调用调 AI 逻辑层，AI 逻辑层通过 SDK 调外部服务和数据层。

**演示**：切到 PPT 上的组件图。

### Q5：你们项目的部署图是怎样的？

**话术**：
我们设计了两套部署方案：

**开发期（当前）**：单机部署
- 前端：Vite dev server，端口 5173
- 后端：FastAPI + Uvicorn，端口 8000
- ChromaDB：嵌入式模式，与后端同进程
- SQLite：本地文件
- LLM：调用外部 API

**生产期（规划）**：容器化分布式部署
- 前端：构建为静态文件，部署在 Nginx
- 后端：Docker 容器，无状态可水平扩展，前置负载均衡
- ChromaDB：独立服务部署，方便后续替换为 Milvus
- SQLite → PostgreSQL：高并发场景升级
- 文件存储：本地磁盘 → 对象存储（OSS/S3）
- 日志监控：集成 Prometheus + Grafana

**演示**：切到 PPT 上的两张部署图对比。

### Q6：你们项目的包图是怎样的？

**话术**：
按职责分层组织：

```
docchat/
├── frontend/       # React 前端项目
├── backend/
│   ├── api/        # FastAPI 路由层
│   ├── services/   # 业务逻辑（document_service, chat_service）
│   ├── rag/        # RAG 核心（retriever, generator, memory）
│   ├── parsers/    # 文档解析器（pdf, docx, md 各一个模块）
```

依赖方向严格遵循：`api → services → rag/parsers → storage`，下层不依赖上层，便于测试和替换。

### Q7：是否有使用虚拟化技术的计划？

**话术**：
有，我们规划在生产部署阶段全面使用容器化：

1. **Docker**：后端服务容器化，确保环境一致性。每个团队成员开发时也用同一个 Dockerfile，避免"在我机器上能跑"问题
2. **Docker Compose**：开发期用 compose 一键拉起整个开发环境（后端 + ChromaDB + 前端）
3. **Kubernetes（远期）**：用户量上来之后，迁移到 K8s 实现自动扩缩容、滚动更新、故障自愈
4. **当前进度**：MVP 阶段优先功能闭环，容器化是 Sprint 4 之后的优化任务，但 Dockerfile 我们已经有初步草稿

### Q8：项目中是否存在潜在的安全风险？你们将如何降低这些风险？

**话术**：
存在多类安全风险，我们用 **STRIDE 模型**梳理并设计了对应方案。这里讲三个最重要的：

1. **数据隔离风险**：用户 A 不应能访问用户 B 的文档。**应对**：在 ChromaDB 中按 user_id 分 collection，所有查询强制带 user_id 过滤；JWT 鉴权确保身份不可冒充
2. **Prompt 注入攻击**：用户可能在上传的文档或提问中植入"忽略上面所有指令"等恶意内容。**应对**：系统 Prompt 与用户输入用结构化分隔；对输出做敏感指令模式过滤；将检索到的文档内容用明确标记包裹，告诉 LLM "以下是参考资料，仅供回答问题"
3. **LLM 数据泄露**：用户文档内容会发送给 LLM 提供商，敏感数据可能被对方记录。**应对**：在用户协议中明确告知；提供"本地模式"选项（用本地 Ollama 跑开源模型）；推荐企业用户选择支持 ZDR（零数据保留）的 API

完整的威胁清单还包括文件上传攻击、API 滥用、传输安全、存储安全、依赖安全、日志审计共 10 类，详见我们的安全设计文档。

**演示**：切到 PPT 上的安全风险表。

### Q10：基于潜在的需求/容量，项目中是否存在性能瓶颈？你们将如何改进？

**话术**：
存在多个瓶颈，按优先级讲三个核心的：

1. **LLM API 延迟（最大瓶颈）**：单次调用 3-8 秒，用户感知最明显
   - **改进**：流式输出（首字响应 < 1 秒）；简单问题路由到小模型（gpt-4o-mini）；相同问题做语义缓存命中
   
2. **文档解析慢**：大 PDF 解析需要 30 秒以上，会阻塞 HTTP 请求
   - **改进**：改为异步任务，前端通过轮询或 WebSocket 接收进度；对相同文件 hash 做去重，避免重复解析

3. **向量检索性能**：数据量超过百万级后会退化
   - **改进**：ChromaDB 默认已用 HNSW 索引，性能尚可；规模再大就迁移到 Milvus/Qdrant，支持分布式与 GPU 加速

性能指标承诺：首字 ≤ 2 秒，完整回答 ≤ 10 秒，10 页 PDF 索引 ≤ 30 秒，支持 100 并发无降级。

**演示**：切到 PPT 上的性能瓶颈表。

## 四、技术速查（团队学习用）

### 4.1 RAG 核心概念三分钟讲懂

**RAG（Retrieval-Augmented Generation）**：让 LLM 在回答之前先从知识库检索相关内容，再基于检索结果生成回答。解决两个核心问题——LLM 知识过时 + LLM 幻觉编造。

**工作流程三阶段**：
1. **索引（离线）**：文档 → 分块 → 向量化 → 存入向量库
2. **检索（在线）**：问题 → 向量化 → 找最相似的 K 个块
3. **生成（在线）**：检索结果 + 问题 → 拼成 Prompt → LLM 生成回答

### 4.2 关键术语

| 术语 | 一句话解释 |
|------|----------|
| **Embedding** | 把文字变成一串数字（向量），意思相近的向量距离也近 |
| **Chunking** | 把长文档切成小段（如 1000 字一段），因为 LLM 上下文有限 |
| **向量数据库** | 专门存向量的数据库（ChromaDB），核心能力是"找最相似的向量" |
| **Top-K 检索** | 从向量库里捞最相似的 K 个段落（一般 K=4） |
| **Prompt 模板** | 把检索到的段落和用户问题按固定格式组装，发给 LLM |
| **Query Rewriting** | 把口语化的追问改写成独立问题，提高检索准确率 |

### 4.3 Query Rewriting 详解（避免被问倒）

**问题场景**：
- 用户第 1 问："这篇论文的作者是谁？"
- 用户第 2 问："**他们**用了什么方法？"

如果直接拿"他们用了什么方法？"去检索，"他们"指代不明，召回率极差。

**Query Rewriting 做什么**：在检索前，先用一次 LLM 调用，输入"对话历史 + 当前问题"，输出一个独立、自包含的问题，比如改写成"论文作者使用了什么方法？"，再用这个去检索。

**实现成本**：LangChain 提供 `create_history_aware_retriever`，几行代码即可，但会多一次 LLM 调用（用便宜的小模型即可）。

**收益**：多轮对话场景下检索准确率显著提升。
